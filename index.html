El Problema

1. El Problema Estratégico

  • Conteos No Confiables: Métodos manuales o barras Infrarrojas (IR) con errores del 10-15% (o más en horas pico), generando datos inutilizables.
  • Frecuencias Ineficientes: Sin datos reales de demanda, no podemos ajustar los headways (tiempo entre buses), resultando en buses vacíos o buses saturados.
  • Auditorías Manuales: Procesos lentos, costosos y propensos a error humano para validar ingresos o subsidios.
  • Experiencia de Usuario Deficiente: Pasajeros sin visibilidad de cuándo llegará el bus o si vendrá lleno, generando frustración.
  • Costos Ocultos: Mantenemos un GPS externo (\$180/mes) redundante y una barra IR ($22,100 CAPEX) que ofrece datos de baja calidad.
La Solución

2. La Solución: Plataforma de Movilidad Inteligente

Proponemos una plataforma integral que resuelve estos problemas con tecnología de alta precisión, proporcionando datos de negocio accionables.

  • QUÉ ES

    Una solución unificada de Conteo de Pasajeros de Alta Precisión (APC) y Telemetría (GPS/4G) en tiempo real.

  • FOTO EJECUTIVA (POR BUS)

    • Cerebro: 1x MDVR Vehicular (Dahua GFWI) con GPS/4G/Wi-Fi integrados.
    • Ascenso (Crítico): 1x Cámara 3D (Visión Estéreo) con 98-99% de precisión.
    • Descenso: 1x Cámara 2D de alta gama con ~95% de precisión.

  • LA CLAVE

    Datos procesados en el bus y enviados por push seguro (HTTPS 443) a nuestro backend, dándonos propiedad total de los datos.

El Valor

3. El Valor de Negocio: De "Contar" a "Gestionar"

Esta no es una actualización de hardware; es una actualización de inteligencia.

Para Finanzas (ROI)

  • Revenue Assurance: Cuadratura de abordajes (99% precisión) vs. boletos o subsidios.
  • Eliminación de Costos: Adiós a auditorías manuales y renta de GPS externo.

Para Operaciones (Eficiencia)

  • Optimización de Headways: Datos reales de demanda por hora y ruta.
  • Despacho Dinámico: Enviar refuerzos solo cuando la métrica `inside` lo indique.
  • Reducción de Dwell Time: Flujo de pasajeros libre, sin frenos (ahorro 8-15s por parada).

Para el Pasajero (Valor)

  • Información en Tiempo Real: Permite crear una App con ETAs precisos.
  • Nivel de Ocupación: Decirle al usuario "Bus Próximo: Lleno" o "Bus Próximo: 50%".

Técnico

11. Arquitectura de Sistema y Flujo de Datos

Diagrama Físico (Por Bus)

[Cámara 3D (Ascenso)] ──(PoE)──► [Mini-switch PoE Vehicular] ──(Uplink)──► [MDVR DHI-MXVR4104-GFWI]
[Cámara 2D (Descenso)] ──(PoE)──► [Mini-switch PoE Vehicular]

Flujo de Datos (Lógico)

  1. Conteo (Push): Cámaras → `Agente Push` (consume CGI `attach`) → Buffer (SQLite) → `HTTPS 443 (Saliente)` → `[Tu API/Backend (Conteo)]`.
  2. GPS (Push): MDVR (Módulo GNSS) → `Protocolo JT808 (0x0200)` → `[Tu API/Backend (GPS Listener)]`.
  3. Gestión (Opcional): MDVR → `(LAN/VPN Taller)` → `[Mobile Center On-Prem]`.

Pilar de Seguridad

  • CERO PUERTOS ABIERTOS EN EL BUS.
  • El MDVR y las cámaras NUNCA deben estar expuestos a Internet.
  • Toda la comunicación es saliente (push) por HTTPS 443 (Conteo) o JT808 (GPS).

Técnico

13. Integración de Datos (1/2): Agente Push y Conteo

El "Agente Push" es la pieza de software (desarrollo propio) que corre en el bus y es responsable de la comunicación segura.

Funciones del Agente

  1. Suscribirse: Conectarse vía HTTP a la cámara (IP local) usando el endpoint de long-polling: 
    GET /cgi-bin/videoStatServer.cgi?action=attach&channel=1&heartbeat=5
    (No usar `getSummary`)
  2. Parsear: Interpretar la respuesta (XML o JSON) para extraer `in`, `out` e `inside`.
  3. Enriquecer: Añadir metadata (bus_id, route_id, door_id, timestamp).
  4. Bufferizar: Guardar eventos en una cola local (SQLite) por si falla el 4G.
  5. Enviar: Hacer un `POST` HTTPS 443 al backend con el evento.
  6. Idempotencia: Generar un `event_id` único (hash) para que el backend pueda descartar duplicados.
  7. Reintentos: Usar backoff exponencial para reenviar eventos fallidos desde la cola.

14. Integración de Datos (2/2): Payload JSON y GPS

Esquema JSON de Evento (Recomendado)

Este es el payload que tu Agente Push debería enviar a tu API.

{
  "device_sn": "DH-3D-ASC-001",
  "bus_id": "BUS-027",
  "route_id": "RT-12",
  "door_id": 1,
  "ts": "2025-11-08T10:32:41-06:00",
  "in": 3,
  "out": 1,
  "inside": 24,
  "rule_id": "ASC_UMBRAL",
  "event_id": "hash(sn|ts|door|in|out)"
}

Protocolo GPS (JT808)

  • El MDVR GFWI soporta nativamente el protocolo JT808 (estándar de telemetría).
  • Se configura en el MDVR para apuntar a la IP/Puerto de tu listener de telemetría.
  • El mensaje principal es 0x0200 (Location Information Report).
  • Esto es más eficiente y directo que usar Mobile Center como intermediario.

16. Implementación: Calibración IA (Crítica)

Una mala calibración invalida toda la inversión. Esto toma 5-10 minutos por puerta.

Pasos (Vía Web UI o SmartPSS)

  1. Activar Plan: AI → Smart Plan → People Counting → ON.
  2. Configurar Regla: People Counting
    • Trazar la línea IN/OUT exactamente en el umbral de la puerta.
    • Realizar 10-20 cruces de prueba (día y noche) para validar.
  3. Ajustes de Video (Noche):
    • Smart IR: Poner en `Bajo` o `Off` (si la luz interior es suficiente).
    • WDR: `ON` (120dB).
    • Anti-Flicker: `60 Hz` (para LATAM).
  4. Ajustes 3D (Cámara 8441X-BV-3D):
    • Asegurar que la altura de montaje (Ej. 2.4m) esté configurada correctamente.
    • Asegurar que el lente sea 2.0 mm (viene de fábrica).
